1. The Problem with Primitive Types in Java

What issues does Java have with primitive types?

  • Primitive types in Java include: byte, short, int, long, float, double, char, boolean.
  • Developers constantly face the risk of data loss (loss of information) when performing type casting between primitive types.
  • For example, converting from int to float may silently lose precision without any warning.
  • To perform safe casting (e.g., int → byte), developers must manually write range checks, which are verbose and error-prone.

Consequence & Solution

Consequence: Code becomes verbose and lacks consistency between primitive types and reference types when using Pattern Matching.

Solution: That is why JEP 530 was introduced in the OpenJDK ecosystem to standardize and solve this problem.

2. JEP 530: Extending Pattern Matching to Primitive Types

  • Definition: JEP 530 (Primitive Types in Patterns, instanceof, and switch) is a language feature currently in its 4th Preview stage (expected in JDK 26).
  • Main Goal: Enhance Pattern Matching by allowing primitive types in all pattern contexts, and extend instanceof and switch to support all primitive types.

Key idea

Make instanceof becomes a safe casting guard, meaning it can check and convert data safely in one step

3. Why JEP 530 Matters

1. Limitation of switch

Pattern matching in switch currently supports only reference types and record patterns (e.g., case Integer i), completely ignoring primitive types.

public class Main { public static void main(String[] args) { int x = 10; switch (x) { case int i -> System.out.println(i); // Compile error: primitive int // does not match pattern } } }

2. Limitation of Record

When decomposing a record, primitive patterns are invariant. You must match double and manually cast to int.

record Point(double x, double y) {} Point p = new Point(3.7, 4.2); if (p instanceof Point(double x, double y)) { int xi = (int) x; // must cast manually int yi = (int) y; }

3. Limitation instanceof

Cannot be used with primitive type patterns. Developers have to do everything manually.

int x = 10; // Does not exist if (x instanceof int i) { System.out.println(i); }

4. How JEP 530 Works

1. Extended Pattern Matching

Primitive type patterns can now be applied to a wider range of match candidates.

2. Top-level Patterns

Primitive patterns are supported at the top level in both instanceof and switch.

3. Enhanced instanceof

instanceof becomes a universal safety guard for all types.

4. Enhanced switch

switch now supports all primitive types, including long, float, double, and boolean.

5. Before vs After: Safer Primitive Casting

Problem: Safely cast from int to byte.

Old Java Style (Before JEP 530)

int a = 200; if (a >= -128 && a <= 127) { byte c = (byte) a; System.out.println(c); } else { System.out.println("Out of range"); } int b = 130; byte c = (byte) b; System.out.println(c); // -126 (Incorrect but no error message)

With JEP 530

int a = 200; if (a instanceof byte c) { System.out.println(c); } else { System.out.println("Out of range"); } int b = 130; if (b instanceof byte c) { System.out.println(c); } else { System.out.println("Conversion unsafe"); }

Result: JEP 530 automatically handles lossy conversion checks and returns false if data loss occurs, eliminating silent bugs

6. Advanced Use Case: Nested Record Patterns

Problem: Parse JSON flexibly while extracting int from double.

Old Java Style

Object json = new JsonObject(Map.of( "age", new JsonNumber(20.9) )); if (json instanceof JsonObject obj) { Object ageField = obj.map().get("age"); if (ageField instanceof JsonNumber num) { double a = num.value(); int age = (int) a; // Manual cast (may result in data loss) System.out.println(age); } }

With JEP 530

Object json = new JsonObject(Map.of( "age", new JsonNumber(20.9) )); if (json instanceof JsonObject(var map) && map.get("age") instanceof JsonNumber(int age)) { // Only run if the conversion is safe. System.out.println(age); } else { System.out.println("Invalid or unsafe age"); }

7. Under the Hood: Conversion Rules & Dominance

  • Exact vs. Inexact Conversions: Safety depends on runtime values.
    • Example: Converting 1000 to byte is inexact → returns false.
  • Unconditionally Exact: The compiler knows the conversion is always safe (e.g., int to double), so no runtime check is needed.
  • Dominance: If one case always matches before another, it call dominance

    ⭢ Helps the compiler detect unreachable code

    • Example: case long dominates case int i

8. Key Benefits of JEP 530

Cleaner Code

Eliminates verbose range-checking logic.

Data Safety

Prevents silent data loss.

Consistency

Unified behavior between primitive and reference types.

9. Limitations and Trade-offs

Backward Compatibility

Stricter dominance rules may cause previously valid switch statements to fail compilation.

long x = 10; switch (x) { case long l -> System.out.println("long: " + l); case int i -> System.out.println("int: " + i); // Error: pattern is dominated by preceding case }

Learning Curve

Developers must adapt to instanceof and switch working with all primitive types.

10. Real-World Applications

Boolean Switches Replace Ternary

Use boolean switch instead of ternary operators for clearer logic

Old style

boolean isPaid = true; boolean isShipped = false; String status = isPaid ? (isShipped ? "Completed" : "Processing") : "Pending";

Modern style

boolean isPaid = true; boolean isShipped = false; String status = switch (isPaid) { case false -> "Pending"; case true -> isShipped ? "Completed" : "Processing"; };

Improved Data Parsing

Very useful for JSON/XML frameworks where input data is often flexible (like numbers that can be int or double) but domain logic needs precise types.

11. What's Next for Pattern Matching in Java

  • New possibilities: In the future, programmers can directly pass constants into Record Patterns like case Box(42) -> ...
